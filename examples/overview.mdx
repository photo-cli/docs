These examples show how photo-cli transforms a folder of photos using the `copy` command. Each example shows the original folder structure, the exact command to run, and the resulting output folder so you can compare what changed and why.

## The test photo set

Every example on this site uses the same starting folder: 17 photos spread across two subfolders. The **Italy album** folder holds 13 photos taken between December 2005 and October 2008, with GPS coordinates pointing to locations in Arezzo and Florence. The **Spain Journey** folder holds 3 photos вЂ” two taken in Madrid in April 2015 and one with no EXIF data at all. Two photos (`DSC_5727.jpg` and `GOPR6742.jpg`) live at the root level, taken in Kenya and the United Kingdom respectively. One photo (`Italy album/IMG_2371.jpg`) has a taken date but no GPS coordinate, and `Spain Journey/IMG_5397.jpg` has neither.

## Archive command example

The `archive` command is different from `copy`: it is designed for long-term, incremental storage. It always organizes photos into a fixed `[year]/[month]/[day]` hierarchy, embeds a SHA1 hash in every file name to prevent duplicates, and records everything in a local SQLite database.

Run the archive command against the same 17-photo test set:

<CodeGroup>
```bash Long form
photo-cli archive \
  --output archive-folder \
  --reverse-geocode OpenStreetMapFoundation \
  --openstreetmap-properties country city town suburb
```

```bash Short form
photo-cli archive -o archive-folder -e 2 -r country city town suburb
```
</CodeGroup>

### What happened

- **Year/month/day hierarchy.** Every photo lands in a `[year]/[month]/[day]` folder derived from its EXIF taken date, regardless of how it was originally organized.
- **SHA1 hash in file names.** Each file is renamed to `yyyy.MM.dd_HH.mm.ss-{sha1}.ext`. The SHA1 hash is computed from the file content, so the name is unique to the exact pixel data.
- **Automatic deduplication.** `DSC_1770.JPG` and `DSC_1770_(same).jpg` are byte-for-byte identical. The archive command copies only one of them and logs a warning. The output contains 16 unique photos from the 17 input files.
- **No-date fallback.** `Spain Journey/IMG_5397.jpg` has no taken date. It is placed in `no-photo-taken-date/` and named with only its SHA1 hash.
- **SQLite database.** After copying, photo-cli writes `photo-cli.sqlite3` into the root of the output folder. This database records the path, taken date, GPS coordinates, reverse-geocoded address, and SHA1 hash for every archived photo. You can query it with any SQLite client.

<Note>
The archive command verifies every copied file by re-computing and comparing SHA1 hashes after the copy completes. If a disk error corrupted a file during the operation, photo-cli will report it immediately.
</Note>
